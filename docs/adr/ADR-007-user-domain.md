# ADR-007: User domain — BIGINT id, UUID habit id retained, Liquibase-only seed, DB-layer `WHERE user_id = ?` ownership

## Status
Accepted

## Context

PBI-012 introduces a `users` table and an owning `user_id` column on `habits`
so that every API endpoint, SQL query, and future Phase 1/2 analytics function
can scope data to a specific user. Four decisions needed an explicit resolution
before the Developer agent could start — the rest of the implementation flows
from them deterministically.

### 1. User ID type — BIGINT serial vs UUID
`habits.id` is UUID (ADR-002) because habits are created client-driven and we
wanted opaque, client-unpredictable identifiers. Users in this PoC are a
different shape:

- There is exactly one seeded user for the entire PoC (`id=1, name='default'`).
- There is no user-creation endpoint (out of scope for PBI-012) and no plan to
  add one until multi-user support lands.
- The user id appears in every subsequent API path (`/users/{userId}/...`) —
  a 19-character UUID in every URL is noise that buys nothing when the
  population is `{1}`.
- Phase 1 already hard-codes `userId: Long` in its public signatures
  (`completionRateByDayOfWeek(userId: Long)`,
  `habitConsistencyRanking(userId: Long)`), which is a structural commitment to
  a numeric id.

### 2. Habit ID type — UUID retained
The Phase 1 brief originally used `streakForHabit(habitId: Long)`. The engineer
decided on 2026-04-22 that `habits.id` stays UUID (the existing schema, ADR-002,
and the entire repository/DTO/codec stack are built on UUID) and the Phase 1
brief has been amended to `streakForHabit(habitId: UUID)` and
`HabitContext.streaks: Map[UUID, Int]`. This ADR records the decision for
traceability; no schema migration of `habits.id` is required by PBI-012.

### 3. Seed data strategy — Liquibase changeset vs Docker Compose init SQL
The `(id=1, name='default')` row needs to exist after a fresh environment is
brought up. Two placement options exist:

- **Docker Compose init SQL** in `infra/db/init/` — runs once at Postgres
  container creation, outside Liquibase's tracking. The existing
  `01_enable_vector.sql` is of this kind (it is correct there because extensions
  must exist before any migration runs and because it is a database-level
  concern, not a table-level one).
- **Liquibase data changeset** — a `--changeset` that runs `INSERT INTO users
  (id, name) VALUES (1, 'default') ON CONFLICT (id) DO NOTHING;` as part of the
  tracked migration history.

### 4. Ownership enforcement — DB-layer `WHERE user_id = ?` vs service-layer lookup
Every habit read must guarantee that a habit belonging to user A is never
returned to user B. Two strategies:

- **DB layer.** Every repository method takes `userId: Long` and appends
  `AND user_id = ?` to the SQL. Ownership is a predicate of the same query that
  fetches the row; there is no separate round-trip.
- **Service layer.** Repository methods stay as they are; the service calls
  `repo.findActiveById(habitId)` and then compares `habit.userId == userId`
  in Scala before proceeding.

---

## Decision

### 1. User ID is `BIGINT` with an explicit literal value seeded (no autoincrement for the seed)

- `users.id` is declared `BIGINT PRIMARY KEY` (no `BIGSERIAL`, no `IDENTITY`).
  Because PBI-012 has no user-creation endpoint, there is no insert path that
  needs a sequence; hard-coding `id=1` for the seed is sufficient and avoids
  binding the schema to a generation strategy before we know what the real
  user-creation flow looks like.
- In Scala the id is `Long` (mapped by Doobie's built-in `Meta[Long]` against
  `BIGINT`).
- `habits.user_id` is declared `BIGINT NOT NULL REFERENCES users(id)`.
- Rationale: `BIGINT` is cheap, human-readable in URLs, matches the Phase 1
  brief's `userId: Long`, and avoids the UUID-in-URL noise for a population of
  one. A future ADR can add a sequence / `BIGSERIAL` (or switch to UUID) when
  user creation is implemented; adding `GENERATED BY DEFAULT AS IDENTITY` later
  is a non-breaking change.

### 2. `habits.id` stays UUID — no schema migration
- No changeset touches `habits.id` or any FK referencing it.
- Phase 1 signatures stay `UUID` for `habitId` (the Phase 1 brief was amended
  by the engineer on 2026-04-22 to match this).
- Rationale: the entire habit codec / repository / DTO / test stack is built on
  UUID; migrating would ripple through every layer for no current-phase benefit.

### 3. Seed placement — Liquibase data changeset (not Docker Compose init SQL)

The `(id=1, name='default')` insert lives in changeset
`004-add-user-id-to-habits.sql` as one of two DML statements in that changeset
(seed first, then backfill `habits.user_id = 1`). Concretely:

- The seed is part of the same Liquibase changeset that depends on it. This
  guarantees ordering by construction — the backfill `UPDATE habits SET
  user_id = 1` cannot run before the parent row exists.
- The insert uses `ON CONFLICT (id) DO NOTHING` so that re-running Liquibase
  against an already-seeded database is a no-op (idempotent at the statement
  level, not just at Liquibase's changelog level).
- The changeset is tagged `failOnError="true"` (Liquibase default) and
  `runOnChange="false"` (default) so a later edit to this changeset would
  trigger Liquibase's checksum guard rather than silently re-running.
- Rationale: Liquibase is already the single source of truth for schema
  (ADR-003). Putting the seed in Liquibase keeps schema and data-prerequisites
  colocated, makes the seed visible in `DATABASECHANGELOG`, makes the seed
  available to Testcontainers integration tests for free (they run Liquibase;
  they do not start a Docker Compose stack), and avoids the cross-tool ordering
  question ("did the container init SQL run before the Liquibase UPDATE?").
  `infra/db/init/` is reserved for true database-level bootstrap (the existing
  `CREATE EXTENSION vector` belongs there; a data row does not).

### 4. Ownership — enforced at the DB layer via `WHERE user_id = ?` on every query

- Every `HabitRepository` method gains a `userId: Long` parameter.
- Every `SELECT`, `UPDATE`, and soft-delete `UPDATE` adds `AND user_id = $userId`
  to its `WHERE` clause.
- `HabitRepository.create(habit: Habit)` continues to take the full `Habit`;
  the `Habit` domain model gains a `userId: Long` field, so the value flows
  through the insert via the existing record.
- `HabitCompletionRepository` does **not** gain a `userId` column or filter.
  Completions reach the user scope transitively: the service calls
  `habitRepo.findActiveById(habitId, userId)` first; if that returns `None`,
  the service returns 404 before touching the completion repository. The
  completion ownership check is therefore "habit owned by user", one round-trip,
  which is exactly what the existing `DefaultHabitCompletionService` does today.
- Ownership mismatch (habit exists but belongs to a different user) returns
  **404 `NotFound`**, not 403 `Forbidden`. This prevents a user from probing
  whether a habit id exists under another user's account.

Rationale:
- **One round-trip.** A service-layer lookup ("fetch habit, then compare
  `habit.userId`") doubles the SQL traffic for every single-row endpoint
  (GET / PUT / DELETE on `/users/{userId}/habits/{habitId}`) with no correctness
  benefit — the DB has to filter anyway.
- **No "forgot the check" class of bug.** With the filter baked into the query,
  there is no code path that can accidentally skip the check. A service-layer
  check can be forgotten; a query that does not compile without `userId` cannot.
- **Query planner support.** The DB can use the (future) `(user_id, created_at
  DESC)` index for listings without an extra filtering step.
- **Matches ADR-005 precedent.** Completion ownership uses the same pattern:
  `DELETE ... WHERE id = ? AND habit_id = ?` enforces ownership inside the
  query rather than in Scala. This ADR extends the same style one level up.

## Consequences

**Easier:**
- Every habit query is a single round-trip and is guaranteed user-scoped by
  the SQL itself — reviewers can verify ownership by reading the `WHERE`
  clause, not the service code.
- The seed is visible in Liquibase's tracking table and runs unchanged in
  Testcontainers specs (they invoke Liquibase directly, not Docker Compose).
- Phase 1's `userId: Long` signatures flow through without type adapters or
  lookups.
- Compact, human-readable URLs (`/users/1/habits/<uuid>`).

**Harder / trade-offs:**
- `users.id` has no generation strategy today. The first PBI that adds a
  user-creation endpoint will need to either switch to `BIGSERIAL` / `GENERATED
  BY DEFAULT AS IDENTITY` or accept client-supplied ids. This is a future
  decision but is cheap (a later changeset can add `ALTER TABLE users ALTER
  COLUMN id ADD GENERATED BY DEFAULT AS IDENTITY (START WITH 1000)`).
- Every `HabitRepository` method signature changes. Every call site (service,
  in-memory test double, Doobie implementation) must be updated. This is a
  mechanical but large diff — PBI-012 acknowledges it in AC-5 and AC-10.
- The in-memory `InMemoryHabitRepository` gains a `userId` parameter on each
  method and must filter its map by it. Its existing tests pass `userId = 1L`.
- Ownership mismatch returning 404 (not 403) trades off "accurate error
  reporting" for "does not leak habit-id existence". Acceptable for the PoC;
  matches industry practice for multi-tenant resource endpoints.

**Locked in:**
- User id is `Long` (`BIGINT` in the DB). Any future plan to switch to UUID
  requires a supersede ADR, a Liquibase migration of both `users.id` and
  `habits.user_id`, and an HTTP routing change (all `/users/{userId}` paths use
  `LongVar`).
- The seed mechanism for reference data is Liquibase. `infra/db/init/` is
  reserved for database-level bootstrap (extensions, roles if ever needed), not
  data rows.
- `HabitRepository` is user-scoped at every method. There is no "admin-level"
  read path that crosses users. If such a path is ever needed, it must be a
  **new method** with a clear name (e.g. `listActiveAcrossAllUsers`), not an
  optional `userId` parameter.
- `habits.id` remains UUID; `users.id` remains BIGINT. The two coexist on
  purpose and the difference is documented here for future readers.
