# PBI-002: Create Habit

## User story
As a user, I want to create a new habit by providing a name, an optional
description, and a frequency, so that I can start tracking it.

## Acceptance criteria
- [ ] `POST /habits` accepts a JSON body with the following fields:
      `name` (string, required), `description` (string, optional),
      `frequency` (string, required — `"daily"` or `"weekly"`).
- [ ] On success the endpoint returns HTTP `201 Created` with a JSON body
      representing the newly created habit, including its server-assigned `id`,
      `created_at`, and `updated_at` timestamps.
- [ ] If `name` is missing or blank, the endpoint returns HTTP `400 Bad Request`
      with a JSON error body containing a human-readable `message` field.
- [ ] If `frequency` is not one of the accepted values, the endpoint returns
      HTTP `400 Bad Request` with a descriptive error message.
- [ ] The created habit is persisted to the `habits` table with `deleted_at` set
      to NULL.
- [ ] An integration test exists that POSTs a valid payload and asserts the
      `201` response and the presence of all expected fields.
- [ ] An integration test exists that POSTs an invalid payload (missing `name`)
      and asserts the `400` response.
- [ ] `./gradlew test` passes.

## Out of scope
- Authentication or user identification.
- Custom frequency values beyond `daily` / `weekly` (schema supports them, but
  the API does not need to validate or reject them in this PBI — that is deferred).
- Any frontend work.

## Notes
- Depends on PBI-001 (domain model and schema must exist first).
- Depends on the framework ADR (http4s vs Play) being resolved before implementation.
- `id` must be a UUID generated by the application layer, not the database.
- Error response shape should be agreed upon before implementation and remain
  consistent across all PBIs — Architect to define a standard `ErrorResponse`
  case class.
